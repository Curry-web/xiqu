# 戏曲唱戏评分算法说明

## 1. 系统概览

| 项目 | 说明 |
| --- | --- |
| 评分目标 | 将用户演唱与指定示范唱段进行声学比对，并结合唱词识别结果生成练功分数 |
| 总分 | 100 分 |
| Python 技术栈 | Python 3.11、FastAPI、Librosa、NumPy、SciPy |
| 音频分析服务 | `scoring-service/app/main.py`、`scoring-service/app/scoring.py` |
| Node.js 接入 | `server/src/services/pythonScoring.js` |
| 综合分计算 | `server/src/services/practiceScoring.js` |
| 外部服务 | 腾讯云口语评测，仅作为唱词识别参考 |
| 当前算法版本 | `python-librosa-pyin-dtw-v1` + `practice-v4-opera-acoustic` |

## 2. 完整评分流程

| 步骤 | 执行模块 | 处理内容 | 输出 |
| --- | --- | --- | --- |
| 1 | Web 前端 | 用户录音或上传音频、视频，选择练习唱段并提交标准唱词 | 原始媒体文件、唱段 ID、唱词 |
| 2 | Node.js 后端 | 校验文件格式并保存上传文件 | 本地上传文件路径 |
| 3 | FFmpeg | 将媒体统一转换为 16kHz、16bit、单声道 WAV | 用户标准 WAV |
| 4 | Python 服务 | 读取示范 WAV 和用户 WAV，提取音高、节奏、颤音、音色、动态等特征 | 声学分项结果 |
| 5 | 腾讯云 | 根据标准唱词评估发音准确度、流畅度和完成度 | 唱词识别参考分 |
| 6 | Node.js 后端 | 按既定权重合成最终练功综合分并生成反馈 | 0～100 分及文字建议 |
| 7 | PostgreSQL | 保存总分、分项分、算法版本和原始评测结果 | 练功记录、我的戏箱数据 |

## 3. 音频预处理参数

| 参数 | 当前设置 | 用途 |
| --- | ---: | --- |
| 采样率 | 16000 Hz | 统一示范与用户录音的时间尺度 |
| 声道 | 单声道 | 避免左右声道差异干扰分析 |
| 帧长度 | 2048 个采样点 | 频谱和音高分析窗口 |
| 帧移 | 320 个采样点 | 每 20ms 生成一帧分析结果 |
| 最低检测音高 | 70 Hz | 覆盖较低人声基频 |
| 最高检测音高 | 900 Hz | 覆盖戏曲高腔和假声区间 |
| 最短音频 | 1 秒 | 低于该长度直接拒绝评分 |
| 静音裁剪 | `top_db=36` | 去除录音前后明显静音 |
| 谐波分离 | `librosa.effects.harmonic` | 尽量突出人声谐波并削弱打击乐、噪声 |

## 4. 分项评分算法

| 页面指标 | 核心算法 | 主要比较内容 | 分项内部公式 |
| --- | --- | --- | --- |
| 唱词识别参考 | 腾讯云口语评测 | 唱词发音、流畅度、完成度 | 发音准确 × 55% + 流畅度 × 30% + 完成度 × 15% |
| 音准旋律 | pYIN + DTW | 基频准确度、旋律升降方向、整体音区 | 绝对音高准确度 × 72% + 旋律走向 × 28% |
| 板眼拖腔 | DTW 对齐路径 | 句长、发声比例、停顿和拖腔时间 | 时长相似 × 45% + 发声比例相似 × 25% + 对齐稳定度 × 30% |
| 颤音转音 | 音高残差 FFT | 颤音速度、幅度、覆盖率及转腔走向 | 颤音相似度 × 68% + 旋律走向 × 32% |
| 音色共鸣 | STFT + MFCC | 明亮度、中频共鸣、频谱包络 | 频谱质心 × 28% + 高频亮度 × 18% + 中频共鸣 × 24% + MFCC × 30% |
| 气息录音 | 周期稳定性 + RMS 包络 | 发声连续性、音量变化、静音、爆音和录音质量 | Python 气息稳定 × 55% + 录音质量 × 45% |

## 5. 最终综合分权重

| 评分维度 | 权重 | 满分贡献 | 设计原因 |
| --- | ---: | ---: | --- |
| 唱词识别参考 | 10% | 10 分 | 戏曲拖腔会降低普通语音识别准确率，因此只作辅助参考 |
| 音准旋律 | 30% | 30 分 | 判断用户是否唱准示范唱段的核心指标 |
| 板眼拖腔 | 20% | 20 分 | 判断句长、节奏、停顿和拖腔是否接近示范 |
| 颤音转音 | 20% | 20 分 | 体现戏曲唱腔中的装饰音和腔尾处理 |
| 音色共鸣 | 10% | 10 分 | 辅助判断声音频谱和共鸣位置，但会受设备影响 |
| 气息录音 | 10% | 10 分 | 判断演唱连续性以及录音是否有效 |
| **合计** | **100%** | **100 分** | 结果四舍五入为整数 |

最终公式：

```text
综合分 =
唱词识别参考 × 10%
+ 音准旋律 × 30%
+ 板眼拖腔 × 20%
+ 颤音转音 × 20%
+ 音色共鸣 × 10%
+ 气息录音 × 10%
```

## 6. 音高与旋律判断

| 项目 | 实现方式 |
| --- | --- |
| 基频提取 | 使用 `librosa.pyin` 检测每帧基频及有声概率 |
| 单位转换 | 将 Hz 转换为音分；1 个半音等于 100 音分，1 个八度等于 1200 音分 |
| 曲线平滑 | 对音高曲线使用中值滤波，降低瞬时误检 |
| 时间对齐 | 使用 DTW 动态时间规整寻找示范与用户音高曲线的最佳对应路径 |
| 八度兼容 | 将八度差折叠到 ±600 音分范围，允许不同性别或音区进行模仿 |
| 整体偏移 | 计算用户相对示范整体偏高或偏低的音分数，并用于反馈 |
| 局部音准 | 去除整体偏移后，计算每个对齐位置的局部音高误差 |
| 旋律走向 | 比较相邻音高变化，判断上行、下行和转腔方向是否一致 |

## 7. 颤音与转音判断

| 项目 | 实现方式 |
| --- | --- |
| 分析窗口 | 64 帧，约 1.28 秒 |
| 窗口步长 | 32 帧，约 0.64 秒 |
| 颤音频率范围 | 3～9 Hz |
| 有效频谱条件 | 3～9 Hz 频带能量占比不低于 32% |
| 有效颤音幅度 | 12～260 音分 |
| 颤音速度分 | 用户颤音频率与示范的接近程度 |
| 颤音幅度分 | 用户上下摆动幅度与示范的接近程度 |
| 颤音覆盖分 | 整句中有效颤音所占比例是否接近示范 |
| 颤音相似度 | 速度 × 35% + 幅度 × 40% + 覆盖率 × 25% |

## 8. 录音质量判断

| 指标 | 判断内容 | 常见扣分原因 |
| --- | --- | --- |
| 时长分 | 是否完整唱完一句 | 音频过短或远长于示范 |
| 有效发声比例 | 非静音演唱帧所占比例 | 前后静音过多、中途长时间无声 |
| 音量分 | RMS 平均音量是否处于合理范围 | 麦克风音量过低或距离过远 |
| 爆音分 | 接近数字满幅的采样比例 | 输入音量过大、贴麦过近 |
| 动态包络 | 用户音量起伏是否接近示范 | 断句、收腔、强弱处理差异较大 |
| 周期稳定性 | pYIN 对基频检测的置信度 | 噪声、气声过多、伴奏过响 |

## 9. 评分结果含义

| 分数范围 | 建议解释 |
| ---: | --- |
| 90～100 | 与示范高度接近，旋律、节奏和唱腔细节较稳定 |
| 80～89 | 整体完成较好，局部转音、拖腔或气息仍可调整 |
| 70～79 | 基本完成唱句，但与示范在部分核心特征上存在明显差异 |
| 60～69 | 能识别出唱段轮廓，音准、节奏或录音质量需要加强 |
| 0～59 | 演唱内容、旋律或录音条件与示范差异较大，建议重新跟唱 |

> 分数表示“与当前示范录音的相似程度”，不是对演员专业水平的绝对评价。

## 10. 服务降级机制

| 情况 | 系统行为 |
| --- | --- |
| Python 服务正常 | 使用 Librosa、pYIN、DTW 完成声学评分 |
| Python 服务超时或不可用 | Node.js 使用内置 JavaScript 声学算法继续评分 |
| Python 和 JavaScript 都无法完成声学比对 | 当前请求进入失败记录，不应只用腾讯云分伪装成完整戏曲评分 |
| 腾讯云调用失败 | 当前请求保存为失败记录，并返回具体错误信息 |
| Python 超时 | `pythonScoring.js` 根据配置中止请求并启用 JavaScript 回退 |

## 11. 当前限制

| 限制 | 影响 | 后续改进方向 |
| --- | --- | --- |
| 参考录音较少 | 目前主要针对配置好的《牡丹亭·游园》唱句 | 为每个剧目、流派和行当建立独立示范库 |
| 单示范模仿评分 | 分数偏向某位示范演员的处理方式 | 同一唱段配置多位专家示范并取最佳或区间得分 |
| 不是真正的板式识别 | “板眼拖腔”主要依赖时间和 DTW 对齐 | 标注板、眼、锣鼓点和关键落音位置 |
| 普通唱词评测不适配拖腔 | 腾讯云可能把延长音、衬字识别为错误 | 使用戏曲唱词强制对齐模型或自建识别模型 |
| 音色受设备影响 | 不同手机、房间和伴奏会改变频谱 | 做设备校准、伴奏分离和响度归一化 |
| 颤音检测为规则算法 | 复杂润腔可能被当作颤音或音高误差 | 建立戏曲润腔标注数据并训练分类模型 |
| 缺少流派知识 | 无法判断梅派、程派等风格是否规范 | 增加剧种、行当、流派条件模型 |
